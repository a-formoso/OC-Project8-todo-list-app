Introduction

The purpose of this documentation is to provide information on an enhanced to-do app.

For lack of a better name, the to-do app in question is simply a task management application built in JavaScript. It allows users to add, edit, delete, or mark tasks as complete. With the filter mechanism, users are also able to see active and completed tasks in a simple and organised way.

Access full application and code (GitHub) repository using the links below:

  • The to-do app: To-do
  • GitHub repo: OC-Project8-todo-list-app
In a small scale, the to-do app implements a JavaScript interpretation of the MVC (Model-View-Controller) architecture. In other words, the app exercises of improved application organisation through a separation of concerns: the isolation of application data (Model) from the visual representation of the model (View), with a third component (Controller) being used to link the other components and manage the logic and user input.

A little more about the to-do app

A fun fact about this to-do app is that it uses (browser) local storage, which means user data is persisted even after the browser is closed/refreshed. Data stored in local storage does not expiry.

Performance. In the Metrics section (above) we're able to see that the first painted content takes 1.6s to appear on the screen (First Contentful Paint). However, as we are dealing with many HTTP requests, the site avarages 3.1s to populate the contents of the page (Speed Index). Consequently, this leads to 8.5s mark in the time a user has to wait until the page becomes interactive or ready to take user input (Time To Interactive) - this is considerably slow compared to our todo app's 1.1s of TTI.

Below, are the opportunities highlighted to optimise performance.
> Eliminate render-blocking resources refers to all URLs (particularly fonts, CSS and jQuery) blocking the first paint of our page. To fix this issue we start by making sure that script elements are placed at the bottom of the page instead; then we should go ahed and identify what's critical and non-critical CSS and JS. In other words, we can go: Audit > (open bottom tab menu) More Options > Coverage. Under Coverage, click the reload icon on the tab submenu to reload and start capturing coverage. This should present a list of URLs with red and blue/green highlights on the right. By opening each URL we can see how much code was used versus how much was loaded. We can reduce the size of our pages by removing the extra code and only shipping the code that we need. Alternatively, we can go on the Network tab and, right-click on a specific resource, and select Block Request URL. This gives us an indication of how a page works without that particular resource.
